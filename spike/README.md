# spike: local MOTIS routing server

A local copy of the routing stack, used for development and for validating
the architecture before deployment. Production runs the same setup on a cloud
VM (see `../DEPLOY.md`); nothing here is required for the live site.

## Run it

```sh
cd spike
chmod +x setup.sh test_queries.sh
./setup.sh              # downloads MOTIS, SL GTFS (1 Trafiklab request), OSM; imports; serves
./setup.sh --serve-only # later runs: skip downloads and import
./test_queries.sh       # in a second terminal: 9 validation queries
```

The server listens on http://localhost:8080. The MOTIS debug UI is served
there too, handy for checking routes by hand.

## What the validation showed

- One-to-all at a 120-minute budget answers in 30 to 40 ms.
- Travel types differ as expected (peak 14,620 reachable stops, weekend
  12,899 from T-Centralen).
- The SL feed includes metro, buses, pendeltag, trams, and ferry services.
- Geocoding handles Swedish addresses and stops well.

## Files

- `.env` holds your Trafiklab API key (never commit it)
- `setup.sh` downloads, imports, and serves
- `test_queries.sh` runs the validation queries; raw JSON lands in `results/`
- `config.yml` is generated by setup.sh

## Notes

- The OSM extract is BBBike's Stockholm metro area. The production server
  uses the same source for now; a full-county clip from Geofabrik's Sweden
  extract is a future upgrade.
- GTFS validity is a rolling window of weeks to months, so representative
  departure dates are always computed relative to today.
- Downloads are atomic (temp file, validate, rename), so an interrupted
  download never poisons the next run.
